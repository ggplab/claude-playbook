# `/new-project`: 새 프로젝트를 일관된 규칙으로 시작하는 스킬

> 새 폴더를 만들 때마다 이름을 고민하고, git을 켜고, `.gitignore`를 챙기던 반복 작업을
> 슬래시 명령 한 번으로 묶은 **Claude Code 커스텀 스킬**입니다.

이 문서는 스킬이 무엇을 하는지 설명하고, 독자가 **자기 환경에 똑같이 만들어 쓸 수 있도록** 단계별로 안내합니다.
같은 폴더의 [`SKILL.md`](./SKILL.md)가 그대로 복사해서 쓰는 템플릿입니다.

---

## 무엇을 해결하나

프로젝트를 새로 시작할 때마다 매번 반복되는 일이 있습니다.

- 폴더 이름을 `my_project`로 할지 `myProject`로 할지 `my-project`로 할지 헷갈린다
- 비슷한 폴더가 이미 있는데 모르고 또 만든다
- `git init`을 깜빡하거나, `.gitignore` 없이 시작해 불필요한 파일이 커밋된다

`/new-project`는 이 흐름을 **하나의 절차**로 고정합니다. 프로젝트 설명만 주면 클로드가
이름 후보를 제안하고 → 기존 폴더와 충돌하는지 확인하고 → 확정된 이름으로 폴더를 만들고
git까지 초기화합니다. 이름을 임의로 정하지 않고 **항상 사용자 확인을 거치는 것**이 핵심입니다.

### 한 줄 사용 예

```
/new-project 농산물 가격 대시보드
```

그러면 클로드가 이렇게 응답합니다(예시):

```
이름 후보 3개입니다:
1. price-dashboard    (입력 직역)
2. agri-price-tracker (목적이 분명)
3. crop-price-board   (간결)

기존 프로젝트 중 'price-'로 시작하는 건 없습니다. 어느 이름으로 만들까요?
```

이름을 고르면 폴더 생성 → `git init` → `.gitignore` → 첫 커밋까지 이어집니다.

## 스킬 파일 해부

Claude Code 스킬은 `SKILL.md` 한 장으로 정의됩니다. 맨 위 **프론트매터**(YAML)와 그 아래 **본문**(클로드에게 주는 지시문)으로 나뉩니다.

```yaml
---
name: new-project
description: 새 프로젝트 디렉토리 생성 전 네이밍 컨벤션 검증 및 이름 추천. ...
argument-hint: "<프로젝트 설명 또는 이름>"
---
```

| 키 | 역할 |
|----|------|
| `name` | 슬래시 명령 이름. 이게 `new-project`면 `/new-project`로 호출된다 |
| `description` | 클로드가 "언제 이 스킬을 꺼낼지" 판단하는 기준. 구체적일수록 정확히 발동한다 |
| `argument-hint` | 입력창에서 보여줄 인자 힌트 (선택) |

본문은 클로드가 따라야 할 **절차**입니다. 이 스킬은 5단계로 짜여 있습니다.

| 단계 | 하는 일 |
|------|---------|
| Step 1 | 현재 프로젝트 목록을 훑어 맥락 파악 |
| Step 2 | 입력을 kebab-case 이름 후보 3개로 변환 |
| Step 3 | 기존 폴더와 이름이 겹치는지 확인 |
| Step 4 | 사용자가 확정하면 폴더 생성 + `git init` + `.gitignore` + 첫 커밋 |
| Step 5 | (선택) Codex 같은 다른 CLI 에이전트에 새 폴더를 신뢰 등록 |

본문에서 쓰는 `$ARGUMENTS`는 사용자가 `/new-project` 뒤에 적은 텍스트로 자동 치환됩니다.

## 따라 만들기 (3분)

### 1. 폴더 만들기

Claude Code는 두 위치에서 스킬을 읽습니다.

- **개인 전역**: `~/.claude/skills/<이름>/SKILL.md`: 모든 프로젝트에서 쓰임
- **프로젝트 한정**: `<프로젝트>/.claude/skills/<이름>/SKILL.md`: 그 레포에서만 쓰임

`/new-project`처럼 어디서나 쓰는 스킬은 전역이 맞습니다.

```bash
mkdir -p ~/.claude/skills/new-project
```

### 2. `SKILL.md` 복사

이 폴더의 [`SKILL.md`](./SKILL.md)를 위 경로에 그대로 넣습니다.

```bash
# 예: 이 레포를 클론한 뒤
cp 02-skills/new-project/SKILL.md ~/.claude/skills/new-project/SKILL.md
```

### 3. 본인 환경에 맞추기

템플릿에서 `★` 표시가 붙은 부분만 바꾸면 됩니다.

- **작업 폴더 경로**: 기본값은 `~/Projects`. 프로젝트를 `~/work`나 `~/dev`에 모은다면 일괄 치환.
- **Step 5 (Codex 등록)**: Codex CLI를 안 쓰면 이 단계를 통째로 지웁니다.
- **`.gitignore`**: 본인 템플릿이 없으면 [github/gitignore](https://github.com/github/gitignore)의 공식 파일을 그대로 써도 됩니다.

### 4. 확인

Claude Code를 다시 띄우고 입력창에 `/`를 쳤을 때 `new-project`가 목록에 보이면 성공입니다.
바로 한번 시켜 보세요.

```
/new-project 테스트용 메모 앱
```

## 왜 이렇게 만드나 (설계 포인트)

- **이름을 클로드가 멋대로 정하지 않는다.** 후보만 제시하고 확정은 사람이 한다. 폴더 이름은
  나중에 바꾸기 번거로우므로 생성 전 한 번 멈추는 게 비용을 줄인다.
- **충돌 체크를 절차에 넣는다.** "비슷한 프로젝트 있나?"를 사람이 매번 기억하지 않아도 되게
  스킬이 대신 묻는다.
- **규칙을 글로 적어 둔다.** kebab-case·금지 문자 같은 컨벤션을 `SKILL.md`에 명시하면,
  클로드가 매번 같은 기준으로 동작한다. 팀이라면 이 파일이 곧 합의된 규칙 문서가 된다.

## 더 확장하기

이 템플릿은 핵심만 남긴 버전입니다. 실제 운영에서는 절차에 단계를 더 붙일 수 있습니다.

- 유형별 `CLAUDE.md`·`.env` 템플릿 자동 복사
- GitHub 레포 자동 생성 (`gh repo create`)
- 폴더에 색 태그/아이콘을 입혀 시각적으로 분류 (운영체제 의존)
- 백엔드가 필요한 프로젝트면 Supabase·Vercel 등 초기 연결

스킬은 결국 "내가 매번 반복하는 절차를 글로 적어 둔 것"입니다. 자주 하는 셋업이 있다면
그걸 Step으로 추가하면 됩니다.
