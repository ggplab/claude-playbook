# CLAUDE.md — 서울 아파트 전세 대시보드

이 파일은 Claude가 이 프로젝트에서 어떻게 일해야 하는지를 정한 규칙서다.
작업 시작 전에 항상 이 파일을 먼저 읽는다.

---

## 1. 이 프로젝트가 뭔가

서울 아파트 실거래 데이터를 바탕으로, **신혼부부가 살 만한 전세집을 찾도록 도와주는 웹 대시보드**를 만들고 계속 운영한다.

- 최종 사용자: 전세를 알아보는 신혼부부 (일반인)
- 대시보드가 답해야 할 질문의 예
  - "예산 안에서 갈 수 있는 동네가 어디인가"
  - "이 단지 전세가가 최근에 오르고 있나 내리고 있나"
  - "같은 값이면 어디가 더 넓은가 / 더 새 건물인가"
- 화면에 나오는 모든 글자는 한국어. 일반인이 읽고 바로 이해할 수 있어야 한다.

### 아직 정해지지 않은 것 (정해지면 이 항목을 지우고 아래에 기록)

- 데이터 출처(어떤 실거래 데이터를 어떻게 가져올지) — 미정
- 사용할 기술(웹페이지를 만드는 방식) — 미정
- 배포 위치(인터넷에 올리는 곳) — 미정

이 세 가지는 **Claude가 임의로 정하지 않는다.** 필요해지는 시점에 선택지 2~3개를 장단점과 함께 한국어로 보여주고, 사용자가 고른 뒤에 진행한다.

---

## 2. 사용자에 대해 (아주 중요)

사용자는 **코드를 전혀 모른다.** 다음을 지킨다.

| 상황 | 이렇게 한다 | 이렇게 하지 않는다 |
|---|---|---|
| 설명할 때 | 한국어로, 화면에서 무엇이 어떻게 보이는지 중심으로 | 함수·라이브러리·빌드 같은 용어를 설명 없이 씀 |
| 확인 요청할 때 | "이렇게 만들까요? A안은 …, B안은 …" 형태로 한국어 질문 | 영어로 묻거나, 코드 조각을 보여주고 판단을 떠넘김 |
| 어려운 말이 꼭 필요할 때 | 한 번은 괄호로 쉬운 말 병기 (예: 배포(인터넷에 올리는 것)) | 그냥 씀 |
| 문제가 생겼을 때 | 무엇이 안 되는지 + 사용자가 할 일 1~2개만 | 에러 메시지 원문만 붙여넣기 |

- 코드 자체를 보여줄 필요는 없다. 사용자가 "코드 보여줘"라고 하기 전엔 결과 화면과 설명만 전달한다.
- 사용자가 직접 명령어를 실행해야 한다면, **한 줄짜리 명령 하나만** 준다. 길거나 줄바꿈이 들어가면 `.sh` 파일로 저장한 뒤 `bash 파일경로` 한 줄만 제시한다.

## 3. 작업 규칙 (반드시 지킬 것)

### 규칙 1 — 산출물은 요청한 파일 1개씩만

- 한 번에 파일 하나만 만든다. 요청하지 않은 보조 파일, 설정 파일, 예제 파일, 문서를 곁들이지 않는다.
- 여러 파일이 꼭 필요하다고 판단되면, **만들기 전에** "이걸 하려면 파일이 2개 필요합니다. ①…, ②… 진행할까요?"라고 한국어로 먼저 묻는다.
- 임시 파일이나 테스트용 파일을 만들었다면 작업 끝에 지운다.
- 이유: 사용자가 코드를 모르기 때문에, 파일이 늘어날수록 무엇이 무엇인지 추적할 수 없게 된다.

### 규칙 2 — 화면을 고쳤으면 브라우저 확인을 안내한다

화면(웹페이지)에 영향을 주는 수정을 한 뒤에는 **반드시** 아래 형식으로 마무리한다.

```
브라우저에서 확인해 주세요.
1. 주소: http://localhost:____  (또는 해당 파일을 더블클릭)
2. 볼 곳: 화면 ____ 부분
3. 정상이면: ____ 이렇게 보입니다
4. 다르면 그대로 알려주세요
```

- 확인 방법을 모호하게("잘 되는지 봐주세요") 쓰지 않는다. 어디를, 무엇이, 어떻게 보이면 성공인지 못 박는다.
- 사용자의 "확인했다" 응답을 받기 전까지 다음 작업으로 넘어가지 않는다.

### 규칙 3 — 데이터를 지어내지 않는다

- 실거래가·시세·단지명 등을 추정하거나 예시로 채워 넣지 않는다.
- 데이터를 못 가져왔으면 "지금은 데이터를 못 가져왔습니다"라고 그대로 보고한다.
- 화면 확인용 임시 데이터가 필요하면, 화면에 **"예시 데이터"라고 눈에 보이게 표시**하고 사용자에게 알린다.

### 규칙 4 — 되돌리기 어려운 일은 먼저 묻는다

파일 삭제, 인터넷 공개(배포), 외부 서비스 전송처럼 되돌리기 어렵거나 남에게 보이는 작업은 실행 전에 한국어로 확인받는다. 파일 수정처럼 되돌릴 수 있는 작업은 바로 진행해도 된다.

## 4. Claude의 작업 방식

공식 프롬프트 가이드(platform.claude.com/docs/ko/build-with-claude/prompt-engineering)의 원칙을 이 프로젝트에 맞춰 옮긴 것.

### 4.1 먼저 확인하고, 추측하지 않는다
- 파일 내용을 열어보지 않고 그 파일에 대해 말하지 않는다.
- 사용자의 요청이 두 가지로 읽히면 임의로 고르지 말고 A안/B안으로 보여주고 기다린다.
- 요청이 한 가지로 읽히면 굳이 되묻지 말고, 어떤 가정으로 진행하는지 한 줄 밝히고 실행한다.

### 4.2 만들기 전에 계획을 보여준다
간단한 수정이 아니라면, 먼저 3~5줄짜리 한국어 계획을 보여주고 승인을 받는다.

```
하려는 것: ____
만들 파일: ____ (1개)
끝나면 확인할 방법: 브라우저에서 ____
진행할까요?
```

### 4.3 과하게 만들지 않는다
- 요청된 것만 만든다. 안 물어본 기능, 설정 옵션, 나중을 위한 구조를 미리 넣지 않는다.
- 화면은 단순하게. 신혼부부가 처음 열었을 때 3초 안에 무엇을 보는 화면인지 알 수 있어야 한다.
- 예외: 디자인 완성도는 아끼지 않는다. 보기 좋고 읽기 쉬운 화면을 만드는 데는 충분히 공을 들인다.

### 4.4 단계마다 검증한다
- 작업을 시작할 때 "이 단계가 끝났음을 어떻게 확인하는가"를 먼저 정한다.
- 화면 작업 → 브라우저 확인 (규칙 2)
- 글자·문구 변경 → 프로젝트 전체에서 옛 문구가 0건인지 검색으로 확인한 뒤 완료 보고
- 안 된 것은 안 됐다고 보고한다. "다 됐습니다"라고 한 뒤 사용자가 열어보고 깨져 있는 상황을 만들지 않는다.

### 4.5 보고는 짧게, 결론부터
- 첫 줄에 결론. 그 아래에 세부.
- 나열할 정보(비교·선택지·상태)는 표로.
- 자화자찬·장황한 요약 금지.

## 5. 화면 만들 때의 기본값

| 항목 | 기준 |
|---|---|
| 언어 | 화면 전체 한국어 |
| 금액 표기 | "3억 2,000만원" 처럼 한국식. 320000000 같은 숫자 나열 금지 |
| 면적 표기 | 제곱미터와 평을 함께 (예: 84㎡ / 25평) |
| 기기 | 휴대폰에서도 깨지지 않게 |
| 색·글꼴 | 하나의 톤으로 통일. 보라색 그라데이션 같은 흔한 조합은 피한다 |
| 빈 화면 | 데이터가 없을 때 "조건에 맞는 매물이 없습니다" 같은 안내 문구를 반드시 넣는다 |

---

## 6. 기록

작업하며 확정된 것은 이 파일에 바로 반영한다.

- 기술 스택이 정해지면 → 1번 "아직 정해지지 않은 것"에서 지우고 여기에 적는다
- 데이터 출처가 정해지면 → 출처 이름과 갱신 주기를 여기에 적는다
- 화면이 늘어나면 → 화면 이름과 각 화면이 답하는 질문을 여기에 한 줄씩 적는다
